# Concepts Inventory — OpenClaw Features Surveyed

Running inventory of OpenClaw concept pages (docs.openclaw.ai/concepts/*) read and triaged for adoption against current Gregor state. Each entry carries an explicit verdict: **ADOPT** (deploy now, with config sketch), **DEFER** (revisit on a concrete trigger — date, event, or condition), or **SKIP** (not for this deployment, with reasoning that won't go stale).

This is not a tutorial — it's a decision log. When new concept pages land in the docs or new releases unlock features, append a new section with verdicts. Do not edit historical verdicts in place; supersede them with a dated follow-up entry so the reasoning trail survives.

Current Gregor baseline as of 2026-04-27:
- Single agent, single Telegram channel, loopback gateway (127.0.0.1:18789)
- Builtin memory engine, Ollama embeddings (`nomic-embed-text:v1.5`, 768-dim)
- 8.4 MB SQLite, 36 chunks, 274 cached embeddings, 7 indexed memory files
- Compaction routes via `openrouter/openai/gpt-4.1-mini`
- Codex OAuth chat path (most Anthropic-only features won't help)
- Dismas down, no second agent in topology right now

## Deferred-Concept Verdicts

### 1. `experimental.sessionMemory` (memory search experimental flag)
**Verdict: DEFER.** The flag indexes prior session transcripts so memory search can surface them, going beyond per-file MEMORY.md retrieval. Cost is documented as "extra storage/indexing" — concrete numbers absent, but for a single Telegram channel with persistent main session this is bounded. Risk profile is low (opt-in, falls back gracefully) but the win is also low for our current usage: Gregor already gets continuity from the auto-memory/MEMORY.md narrative layer plus a stable workspace, and the most recent breakage (workspace drift, Apr 27) was an identity-load problem, not a search-recall problem. No urgent capability gap. **Revisit trigger:** the first time we observe Gregor failing to recall a fact from a prior session that was clearly captured in the live transcript but not promoted to MEMORY.md (i.e., a recall miss, not a write miss). At that point flip the flag and measure index growth on the 8.4 MB SQLite.

### 2. Honcho memory engine
**Verdict: SKIP (for now, with revisit trigger).** Honcho is an external AI-native memory service (managed at api.honcho.dev or self-hosted) that auto-builds user/agent profiles, adds semantic search over distilled observations, and tracks parent/child agent relationships. Three reasons it's wrong for Gregor today: (1) **vendor/network dependency** conflicts with the loopback-only, supply-chain-locked posture — managed mode is a third-party data egress, self-hosted mode is another service to harden; (2) **no multi-agent topology** — parent/child tracking is the headline differentiator and we have one agent; (3) **migration cost** — we'd be moving off a working builtin engine (recently re-indexed, embeddings cached) to chase capabilities we don't currently need. **Revisit trigger:** if/when Dismas comes back AND we want a shared user-profile across both bots, Honcho self-hosted (not managed) becomes worth a re-eval. Also revisit if MEMORY.md's manual user-profile sections become a real maintenance burden (>30 min/month).

### 3. QMD memory engine
**Verdict: DEFER.** QMD is a local-first sidecar (BM25 + vector + reranking, ~2 GB GGUF model, local SQLite with extensions) that adds reranking, query expansion, external-directory indexing, and session transcript indexing — all without external services. **Important correction to the bead's framing:** per docs, QMD does **not** cross agent boundaries — each agent has its own QMD home under `~/.openclaw/agents/<id>/qmd/`, and session indexing is per-agent. Cross-agent transcript search is not a free QMD feature; it would require us to point multiple agents at a shared index, which the docs don't describe as supported. So the original "cross-agent" lure is invalidated. What QMD does still offer Gregor: reranking (real recall improvement on a small corpus is dubious — we have 36 chunks) and external-directory indexing (could index the repo's `Reference/` and `GUIDE.md` so Gregor answers ops questions from primary sources). The 2 GB model + native deps + per-agent isolation is a non-trivial install. **Revisit trigger:** when memory grows past ~200 MB SQLite OR when we explicitly want Gregor to answer from `Reference/*.md` without copy-pasting into MEMORY.md. File a bead before installing.

### 4. Dreaming (background memory consolidation)
**Verdict: DEFER (lean toward ADOPT, but not yet).** Dreaming is opt-in three-phase background consolidation (Light/Deep/REM) that promotes high-signal short-term memories to durable MEMORY.md via a weighted ranking (frequency 24%, relevance 30%, query diversity 15%, recency 15%, consolidation 10%, conceptual richness 6%) on a cron (default `0 3 * * *`). It writes a human-readable DREAMS.md audit trail and uses one background LLM turn per phase against the default runtime model. The blocker for Gregor today is exactly that — the default runtime model is Codex OAuth (`openai-codex/...`), and per the lesson we just learned with compaction, **OAuth providers cannot reliably do background invocations across separate processes.** Dreaming would either fail silently or thrash auth. Before adopting, we'd need to pin Dreaming's model to a key-based provider (likely OpenRouter Haiku, mirroring our compaction fix), and we'd need to confirm that overlap with the existing auto-memory layer (MEMORY.md narrative) doesn't cause double-promotion or contradiction. **Revisit trigger:** after one calendar week of compaction running cleanly on the canonical config (i.e., on or after 2026-05-04), AND once we've identified the right `dreaming.model` key form. Adoption then is small: cron + OpenRouter Haiku, ~$0.10-0.30/month, plus a one-week review of DREAMS.md to verify quality.

### 3a. QMD cross-agent transcript search (bead 5rs follow-up, 2026-04-27)

**Verdict: NOT SUPPORTED via QMD — confirmed.** The original QMD entry above flagged this as an open question; bead 5rs resolved it. Per `docs.openclaw.ai/concepts/memory-qmd`, every agent gets `~/.openclaw/agents/<id>/qmd/` and session transcripts export to `~/.openclaw/agents/<id>/qmd/sessions/` — both paths are agent-scoped with no documented override. There is no `extraCollections` or shared-collection feature in the QMD config surface (`memory.qmd.paths`, `memory.qmd.sessions.enabled`, `memory.qmd.scope`, `memory.qmd.limits.timeoutMs`). **Workaround for shared READ-ONLY corpus** (not transcripts): point multiple agents' `memory.qmd.paths` at the same external directory tree (e.g., a shared `Reference/`). Session-transcript sharing would require either an undocumented config override or a different memory-engine architecture (Honcho parent/child, custom shared SQLite). **Revisit trigger:** if Dismas returns AND we need shared session-transcript search, file a fresh bead — the answer today is "not via QMD."

### 5. Active memory
**Verdict: SKIP.** Active memory is a plugin that injects a blocking memory sub-agent before the main reply, pre-emptively surfacing context instead of waiting for the main agent to call `memory_search`. Cost: one extra inference call per user message in eligible sessions; mitigation is using a fast fallback model. It only runs in interactive persistent chat. Two reasons it's wrong for Gregor: (1) **latency budget** — Telegram users tolerate ~2-5s; an extra blocking inference, even on Cerebras/Gemini Flash, eats that budget for a benefit that's mostly a UX polish on a bot that already has a stable persona and explicit tooling for recall; (2) **Codex OAuth fallback story** is unclear — the docs imply you want a fast non-OAuth model for the sub-agent, which means another paid API key path. The capability gap this fills (reactive vs preventive memory recall) is not one Gregor's users have complained about. **Revisit trigger:** none. If a future user-facing pattern emerges where Gregor consistently misses context that was clearly in memory, file a new bead and re-open this verdict — but the default stance is no.

### 6. Sub-agents (`agents.defaults.subagents`)
**Verdict: DEFER (lean toward SKIP for current Gregor topology, ADOPT-candidate for the future agent-pack arsenal).** Sub-agents (`https://docs.openclaw.ai/tools/subagents.md`) parallelize slow work without blocking the parent run, run in isolated context (default) or fork-from-parent context, support orchestrator patterns at `maxSpawnDepth: 2`, and resolve auth per-agent-id with main-agent profiles merged in as fallback. Defaults: `maxSpawnDepth: 1`, `maxChildrenPerAgent: 5`, `maxConcurrent: 8`, `runTimeoutSeconds: 900`, context mode `isolated` (lower token use). Each child gets its own context + token budget. Three reasons it's wrong for Gregor today: **(1) workload mismatch** — Gregor's primary path is interactive Telegram, not parallel batch; the spawn-without-blocking benefit doesn't apply to a single-turn-at-a-time chat; **(2) OAuth uncertainty** — sub-agents inherit auth via per-id resolution merged with main fallback. The docs note "no explicit prohibition" of OAuth, but per the durable lesson from compaction (KNOWN-BUGS #6) Codex OAuth is unreliable for *separate-process invocations*, which is exactly what a sub-agent run is. Until proven otherwise, treat OAuth-only chat as a sub-agent footgun the same way we treat it for compaction; **(3) chassis-pattern break** — sub-agents do NOT inherit `SOUL.md`, `IDENTITY.md`, or `USER.md` (only `AGENTS.md` + `TOOLS.md`). The AtlasForge identity layer is one of our chassis pillars; a sub-agent runs without it, producing inconsistent persona for any user-visible output. **Revisit trigger:** when we deploy the agent-pack arsenal (per `Plans/next-session-agent-packs.md`) AND have at least one vertical where (a) the workload genuinely benefits from parallel sub-agent spawning (research/scrape/fan-out tasks), (b) a non-OAuth API-key model can be pinned for the sub-agent (`agents.defaults.subagents.model`), and (c) the vertical's identity layer can tolerate the sub-agent context mode (or we accept anonymous workers under an orchestrator). Until those three conditions co-occur, leave `agents.subagents: []` empty.

**Surprise worth filing:** Sub-agents do NOT inherit IDENTITY/SOUL/USER files — only AGENTS.md and TOOLS.md. This means orchestrator patterns lose persona-grounded behavior in workers, which is fine for technical fan-out but breaks user-facing voice consistency. Worth flagging for the agent-pack design phase.

### 7. ACPx — Agent Client Protocol adapter (verified 2026-05-07)

**Verdict: ADOPT inside any agent that needs to delegate coding work, NOT as multi-bot orchestration.** ACPx is the OpenClaw adapter for Zed Industries' Agent Client Protocol — JSON-RPC over stdio, IDE↔coding-agent spec. It lets one OpenClaw agent spawn external coding harnesses (Claude Code, Codex, Cursor, Copilot, Droid, Gemini CLI, OpenCode, Pi) as headless coder sub-processes scoped per repo with persistent multi-turn sessions, named workstreams (`-s backend`, `-s frontend`), and prompt queueing. Live config surface: `acp.dispatch.enabled`, `acp.allowedAgents`, `acp.runtime.ttlMinutes` (default 120), `acp.runtime.maxConcurrentSessions`, per-agent mapping via `agents.list[].runtime.acp.agent`. Gregor's current config has `allowedAgents: ["codex"]` — exactly the intended use case (let main delegate to Codex when it needs to write code).

**It is NOT** a multi-persona orchestration substrate. The phrase "persistent coworker chat-facing personas" does not appear anywhere in `docs.openclaw.ai/tools/acp-agents`. Spec scope is IDE↔coding-agent, not agent↔agent. The native answer for chat-facing multi-agent on one host is `agents.list[]` (see entry 8).

**Adoption path:** Vesalius (V1 Dev/code-copilot) is the natural ACPx user — let Vesalius spawn Codex sessions for coding tasks. Other bots leave the existing `acp.allowedAgents: ["codex"]` shape in place (it's harmless when unused). **Do NOT** broaden `allowedAgents` until a concrete need surfaces; ACP delegation has a known observability bug with `agents.list[]` (GitHub issue #32804: "ACP sessions invisible to sessions_list when agents.list is configured").

**Sources:** `docs.openclaw.ai/tools/acp-agents`, `docs.openclaw.ai/tools/subagents` (frontmatter explicitly distinguishes `runtime: "subagent"` vs `"acp"` vs `"codex"`), `agentclientprotocol.com` (Zed spec), `github.com/openclaw/acpx`.

### 8. `agents.list[]` — Native multi-agent on one gateway (verified 2026-05-07)

**Verdict: DEFER (Option B is real but the 5-bot pack already chose Option A).** Per `docs.openclaw.ai/concepts/multi-agent`: *"The Gateway can host one agent (default) or many agents side-by-side ... each with its own workspace, state directory (agentDir), and session history — plus multiple channel accounts."* `bindings[].match` routes by `channel`, `accountId`, `peer`, `guildId`, roles. This IS the OpenClaw-native answer to "5 personas on one host" — a real Option B against the current 5-Linux-user pack design (Option A) and a Matrix-substrate alternative (Option C).

**Why not adopt now:** the 5-bot pack design (`VERTICAL-AGENTS.md`) chose Option A for blast-radius isolation — Dismas's TG-only allowlist=[Marius] scoping exists precisely because root-tier ops blast radius is load-bearing. Putting Dismas + Vesalius (dev-paste exposure) + Aldine (publishing) on one gateway process collapses the isolation that scoped Dismas in the first place. Plus the Telegram channel plugin's intra-account dispatch story (one bot token routing to N personas via `@aldine ...`) is undocumented; the LumaDock multi-agent tutorial routes via separate `accountId` per agent — i.e. one bot token per persona, which is what Option A already does the slow way.

**Revisit trigger:** if/when (a) Dismas's blast-radius scoping is loosened or moved to a dedicated separate VPS anyway, AND (b) Telegram intra-account dispatch is documented as supported, AND (c) ops cost of 5 systemd units becomes a real maintenance burden. None of those conditions hold today.

**Sources:** `docs.openclaw.ai/concepts/multi-agent`, lumadock.com multi-agent setup tutorial, GitHub issue #32804 (visibility bug to track if Option B is ever adopted).

### 9. Matrix as orchestration chat substrate (verified 2026-05-07)

**Verdict: DEFER (Option C — adds capability Option A can't deliver, but only if the operator wants it).** Matrix (self-hosted Synapse on the same VPS, federation disabled) is a real Option C against A/B for the multi-bot question. It uniquely delivers: (a) persistent thread-bound subagent sessions (Telegram extension lacks the hook in v2026.5.6 — KNOWN-BUGS #13), (b) bot-to-bot fleet chatter via `allowBots: true` + Application Service API, (c) E2EE for the conversation history (Megolm/Olm), (d) per-room `systemPrompt` / `skills` / `tools` config that gives natural per-bot working surfaces. The OpenClaw Matrix extension ships `subagent-hooks-api.js` + `thread-binding-api.js` + `thread-bindings-runtime.js` — Telegram doesn't.

**Why defer:** adding Synapse adds operational surface (another self-hosted service, federation considerations, mobile client UX swap from Telegram to Element). The current pack runs fine on Telegram for the chat surface; the Matrix capabilities are nice-to-haves until concrete need arises.

**Revisit trigger:** the first time one of (a) you actually want a persistent thread-bound child subagent session (e.g. "let me keep talking to this research worker"), (b) Dismas-related conversation content needs E2EE-on-the-wire that Telegram can't provide, OR (c) bot-to-bot routing across the pack becomes a thing you keep needing to do manually. Until then, Telegram + bundled `:18789/` Control UI for single-bot views + `builderz-labs/mission-control` (when fleet boots) covers the surface.

**Sources:** `docs.openclaw.ai/channels/matrix`, OpenClaw Matrix extension files (`subagent-hooks-api.js` etc., not present in Telegram extension), `agentclientprotocol.com` (rule out — different protocol), `github.com/zscole/matrix-agents` (early PoC).

## Surprises Worth Filing

- **QMD does NOT cross agent boundaries.** The bead description hinted at QMD enabling "cross-agent transcript search"; the docs flatly say sessions index per-agent and each agent has its own QMD home. This invalidates one of the strongest motivations for adopting QMD in a multi-bot future. Worth a follow-up bead to research whether a shared QMD home is configurable, or whether cross-agent search requires a different architecture entirely (e.g., Honcho's parent/child model, or a custom shared SQLite with both agents reading).
- **Dreaming + Codex OAuth = same compaction footgun.** Any background-LLM feature that uses `agents.defaults.model` will inherit the OAuth-cross-process unreliability we just fixed for compaction. This generalizes: every future "background AI" feature in OpenClaw needs an explicit API-key model override before it'll work on Gregor. Worth a single durable note (CLAUDE.md or KNOWN-BUGS.md) so we don't re-learn it per feature.
- **ACPx ≠ orchestration.** Easy to misread the plugin name as "agent control protocol" → "fleet orchestration." It's not. ACPx is an external coding-harness adapter (Zed IDE↔coder spec). The OpenClaw-native multi-agent answer is `agents.list[]`, not ACPx. Documenting this here so future-us doesn't re-litigate it.
